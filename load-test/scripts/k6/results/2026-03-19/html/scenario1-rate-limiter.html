분석 및 원인

Phase 3 (Recovery)에서 429 지속 — 트래픽 감소 후에도 복구 안 됨

  1. 1초 고정 윈도 카운터가 리셋되지 않음 (evictExpiredCounters 스케줄러 미동작)확인: LocalRateLimiter.java — @EnableScheduling이 RateLimiterConfig에 선언되어 있는지 확인
  2. Phase 2 → Phase 3 전환 시점에 잔여 요청이 같은 윈도에 걸림확인: 정상 동작이라면 1~2건의 429는 윈도 경계 문제로 허용 범위 (threshold: <5%)

429 응답 P95 125.7ms — 목표(<50ms) 초과

  1. SCG Netty event loop이 포화 상태 — 429를 반환하는 것 자체가 지연됨확인: Grafana에서 reactor.netty.* 메트릭 확인. scg-app CPU 사용률 점검
  2. Rate Limiter 이전 필터 체인(Correlation → AccessLog → InternalBlock → Sanitize → JWT)에서 병목확인: Jaeger span에서 JWT 검증 시간 확인. JWT 파싱이 CPU-bound라 과부하 시 지연 가능
  3. GC pause가 응답 시간에 영향확인: Grafana → jvm.gc.pause 메트릭에서 테스트 시간대 GC pause 확인
